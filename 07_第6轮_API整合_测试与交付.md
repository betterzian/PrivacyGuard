# 第 6 轮：API 整合、测试与交付

## 可直接粘贴到 Cursor 的简短提示词

请在不破坏既有模块边界的前提下，完成 **API facade 整合、配置装配、样例数据、测试补全、README 交付说明**。  
这一轮的重点是“把项目交付为一个可运行、可理解、可扩展的框架”，而不是继续发明新模块。

---

## 本轮目标

让项目达到一个“框架搭建完毕”的状态：

- `PrivacyGuardFacade.sanitize()` 可调用；
- `PrivacyGuardFacade.restore()` 可调用；
- 可通过配置切换 detector / decision / storage 实现；
- 有 sample data、sample config、sample usage；
- 测试可运行；
- README 能让其他开发者快速接手。

---

## 必须遵守的约束

1. 不要再新增核心模块；
2. 只做整合、补文档、补测试、修接口不一致问题；
3. 若发现前几轮设计冲突，优先最小修正，而不是大重构；
4. 所有默认配置都要能在“无真实模型”条件下跑通基础示例；
5. 交付文档必须明确哪些能力是真实实现，哪些是占位/可插拔点。

## facade 最终要求

`PrivacyGuardFacade` 需要真正串起：

### sanitize
- 入参：`SanitizeRequest`
- 调用：`sanitize_pipeline`
- 出参：`SanitizeResponse`

### restore
- 入参：`RestoreRequest`
- 调用：`restore_pipeline`
- 出参：`RestoreResponse`

### 其他要求
- 初始化时接收配置对象或配置路径
- 通过 factories 装配默认实现
- 对配置错误给出清晰异常

---

## 模式切换要求

至少支持以下切换：

### PII Detector
- `rule_based`
- `rule_ner_based`

### Decision Engine
- `label_only`
- `label_persona_mixed`
- `de_model`

### Mapping Store
- `in_memory`
- `json`

### Persona Repository
- `json`

注意：
- 切换机制必须在配置层完成；
- 业务层不得硬编码具体实现类。

## README 交付要求

README 至少包含：

1. 项目简介
2. 核心能力
3. 当前支持的模式
4. 目录结构
5. 安装方式
6. 最小运行示例
7. `sanitize -> restore` 调用示例
8. 当前限制
9. 后续扩展方向

### 特别要求
在 README 里明确写出：
- 当前 `de_model` 是规则占位版；
- OCR / GLiNER 是否依赖真实模型环境；
- 图像重绘是最小可行实现而非最终效果。

---

## 测试补全要求

至少保证：

### facade_sanitize
- 能从默认配置启动
- 能返回结构化 `SanitizeResponse`

### facade_restore
- 能根据已有 mapping 恢复文本

### mode_switching
- 切换不同 detector / decision mode 不会破坏装配

### 建议额外补充
- repository 配置缺失时报错
- 无 persona 数据时的降级行为
- detector 外部依赖缺失时的降级行为
